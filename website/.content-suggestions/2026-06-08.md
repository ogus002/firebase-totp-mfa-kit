## Proposed Improvement

The "Install" section says the CLI copies files into `app/(auth)/mfa-enroll`, which is an App Router path. But the page title and intro both claim support for **Pages Router** too, and the command uses `--area /admin` without showing what the resulting route actually looks like. A developer using Pages Router will be confused about where files land.

The fix adds a concise note immediately after the install command that clarifies the output path for both routers, so the reader knows exactly what was installed without having to run the command first and inspect their filesystem.

### Why this change

| Problem | Detail |
|---|---|
| **Stale/misleading path** | The prose hard-codes `app/(auth)/mfa-enroll` but the command uses `--area /admin`, so the actual output path would be `app/admin/mfa-enroll`, not what the sentence says. |
| **Missing Pages Router path** | The intro promises Pages Router support; the section never shows where files land on that router. |
| **`--area` flag left unexplained** | Readers don't know whether `--area` changes the filesystem path, just the URL, or both. The note clarifies it controls the URL prefix (and therefore the directory name) in both router modes. |
